# Local Model Integration Roadmap

## Vision

Add a lightweight local model (Phi-3, Llama 3, Mistral via Ollama) that handles simple tasks instantly while delegating complex coding work to Copilot. The user experience stays the same — speak naturally, get a spoken answer — but simple requests resolve in under a second instead of waiting 10+ seconds for a Copilot round-trip.

## Architecture

```
User prompt
    │
    ▼
Command detector (keyword matching, fast)
    │
    ├─ known command → Local model / direct execution
    └─ everything else → Copilot (existing flow)
```

The local model is NOT a router that classifies every prompt. It only activates on explicit command patterns. Everything ambiguous goes to Copilot. This avoids routing latency and misclassification.

---

## Quick commands (no model needed)

These are pattern-matched and executed directly, no LLM involved.

- **Time/date** — "What time is it?", "What day is it?"
- **Git status** — "Which branch am I on?", "Any uncommitted changes?"
- **System info** — "How much disk space is left?", "What's my IP?"
- **Proxy control** — "Start new session", "What's your wake word?"

Implementation: keyword matching in `main.py` before the prompt reaches Copilot. Response is generated from a template and sent directly to TTS.

---

## Local model features

These use a local LLM for natural language generation.

### Activity summarization

When the user asks "What's happening?" or "Give me an update" during a Copilot turn:
- Collect recent tool calls, thoughts, and partial text from the bridge event buffer
- Feed them to the local model with a prompt like: "Summarize what the coding agent has been doing based on these events"
- Speak the summary without interrupting Copilot

### Conversation memory

- "What did you just say?" — replay or rephrase the last Copilot response
- "What did I ask?" — repeat the last user prompt
- "Summarize our conversation" — local model summarizes the turn history

### Response rephrasing

- "Say that again simpler" — local model rephrases the last Copilot response in plainer language
- "Give me just the key points" — local model extracts bullet points from a long response
- "Explain that like I'm a junior dev" — local model adjusts the complexity level

### Smart acknowledgments

Replace static wake sounds with context-aware spoken acknowledgments generated by the local model:
- First interaction: "Hey, good morning. What are you working on?"
- After a long session: "Still here. What's next?"
- After an error: "Something went wrong last time. Want to try again?"

The local model sees the conversation history and time of day to generate natural greetings.

## Tool execution (MCP)

The local model can call MCP tools for tasks that don't need Copilot's code intelligence.

### Music/media control
- "Play Master of Puppets on Spotify"
- "Pause", "Skip", "What's playing?"
- Requires: Spotify MCP server

### Browser/web
- "Open the GitHub PR for this branch"
- "Search Stack Overflow for Python asyncio queue"
- Requires: Browser MCP server

### Notifications/communication
- "Read my latest Slack messages"
- "Send a message to the team channel: deploy is done"
- Requires: Slack MCP server

### Deployment/infra
- "Deploy staging"
- "What's the status of the last deploy?"
- Requires: Custom MCP server per infra

## Copilot management

The local model can manage the Copilot session on behalf of the user.

- **Auto-retry on failure** — If Copilot errors out, the local model detects it and offers to retry: "Copilot hit an error. Want me to try again?"
- **Prompt refinement** — If Copilot's response seems off-topic, the local model can suggest: "That didn't seem right. Want me to rephrase your question?"
- **Session health** — Monitor Copilot response times and warn: "Copilot is responding slowly. Might be under load."
- **Context management** — "Forget everything about the autograder" — local model sends a session reset with selective context clearing instructions.

---

## Natural response post-processing

Run Copilot's raw text through the local model before TTS to improve spoken output.

- **Strip code artifacts** — Remove markdown, code fences, bullet markers, URLs that Copilot includes despite instructions not to
- **Pronunciation hints** — Convert technical terms, acronyms, and camelCase into speakable forms ("API" → "A P I", "asyncio" → "async I O")
- **Length control** — If Copilot returns a 500-word response, the local model condenses it to the key points for voice delivery
- **Tone matching** — Ensure the response sounds conversational, not like a written document being read aloud

## Implementation approach

### Phase 1: Quick commands (no model)
Pattern matching for time, git, system info. Direct execution, template responses. Zero new dependencies.

### Phase 2: Ollama integration
Add Ollama as an optional local model backend. Activity summarization and conversation memory. Requires Ollama installed locally.

### Phase 3: Response post-processing
Run Copilot output through the local model before TTS. Strip artifacts, improve spoken quality.

### Phase 4: MCP tool execution
Local model calls MCP tools for non-coding tasks. Spotify, browser, Slack.

### Phase 5: Smart acknowledgments and Copilot management
Context-aware greetings, auto-retry, prompt refinement, session health monitoring.

---

## Requirements

- Ollama (or compatible local inference server) for Phases 2+
- ~4GB RAM for a 7B parameter model
- MCP servers for specific tool integrations (Phase 4)
- Quick commands (Phase 1) have zero additional dependencies
